Step 3 of 6 · In Progress
Install Wazuh Agent
Install the Wazuh HIDS agent on all B1 Registries servers. The agent streams security events to the B3 SIEM. Required by SIEM-35 — brute-force and anomalous access detection.
🎯 What this step does
Installs a lightweight agent (~50MB) that monitors system logs, file integrity, and network activity on your servers. Events are encrypted with your mTLS certificate and streamed to Wazuh Manager at 10.0.1.10:1514. Your bundle data is isolated in the B1-Registries agent group.
Select your operating system:
bash · Ubuntu 22.04 LTS
# 1. Add Wazuh repository curl -s https://packages.wazuh.com/key/GPG-KEY-WAZUH \ | gpg --dearmor -o /usr/share/keyrings/wazuh.gpg echo "deb [signed-by=/usr/share/keyrings/wazuh.gpg] \ https://packages.wazuh.com/4.x/apt/ stable main" \ | tee /etc/apt/sources.list.d/wazuh.list # 2. Install agent — pre-configured for B3 SIEM apt-get update && apt-get install -y wazuh-agent=4.7.5-1 # 3. Configure and start (your Bundle ID pre-filled) WAZUH_MANAGER=10.0.1.10 \ WAZUH_AGENT_GROUP=B1-Registries \ WAZUH_AGENT_NAME=b1-registries-prod-01 \ systemctl enable --now wazuh-agent

Step 4 of 6
Configure Prometheus Scraping
Expose a /metrics endpoint on your B1 servers. B3 Prometheus will scrape it every 15 seconds. Required by APM-22 — application performance monitoring for all DHP bundles.
🎯 What to expose
Add the Prometheus client library to your B1 application. Expose these metric families: http_requests_total, http_request_duration_seconds, process_cpu_seconds_total, process_resident_memory_bytes. Port: 9090. Path: /metrics.
yaml — prometheus.yml scrape config to add
scrape_configs: - job_name: 'b1-registries' scheme: 'https' tls_config: cert_file: '/etc/b3-certs/b1-client.crt' key_file: '/etc/b3-certs/b1-client.key' ca_file: '/etc/b3-certs/moh-root-ca.crt' static_configs: - targets: - '10.1.0.45:9090' - '10.1.0.46:9090' scrape_interval: '15s' labels: bundle: 'B1-Registries' bundle_id: 'LKA-B1-001'

Step 5 of 6
Test AuditEvent POST
Send a test FHIR R4 AuditEvent to the B3 Audit Repository. This confirms your mTLS certificate is accepted, your FHIR client is correctly configured, and events are being SHA-256 chained. Required by AR-43.
⚠ Use the TEST purposeOfEvent code
In production, every AuditEvent must carry the correct purposeOfEvent code (ACCESS, CLINICAL, SECOPS, etc.) per PDPA §29. For this onboarding test, use TEST — these events are automatically excluded from compliance reports.
bash — FHIR AuditEvent POST test
curl -X POST https://audit.b3.moh.gov.lk/fhir/r4/AuditEvent \ --cert /etc/b3-certs/b1-client.crt \ --key /etc/b3-certs/b1-client.key \ --cacert /etc/b3-certs/moh-root-ca.crt \ -H "Content-Type: application/fhir+json" \ -d '{ "resourceType": "AuditEvent", "type": { "system": "http://terminology.hl7.org/CodeSystem/audit-event-type", "code": "110100", "display": "Application Activity" }, "action": "E", "recorded": "2026-07-12T07:00:00Z", "outcome": "0", "agent": [{ "requestor": true, "network": { "address": "10.1.0.45", "type": "2" }}], "source": { "site": "LKA-B1-001", "observer": { "display": "B1 Registries Onboarding Test" }}, "purposeOfEvent": [{ "coding": [{ "code": "TEST" }]}] }' # Expected: HTTP 201 Created # Response will include: id, sha256-chain extension, chain position

Verify NTP Synchronisation
